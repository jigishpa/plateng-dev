Architecture

A single EKS management cluster provisions and manages workload clusters on both AWS and GCP.

Zero Static Credentials

IRSA for AWS, Workload Identity Federation for GCP. No long-lived secrets anywhere.

GitOps-Native

All changes flow through Git. ArgoCD reconciles desired state automatically.

Cloud-Portable

Same claim API provisions EKS or GKE. Crossplane compositions abstract cloud differences.

Observable

Full LGTM stack: Prometheus, Grafana, Loki, Tempo. Dashboards and alerts from day one.

Secure by Default

Kyverno policies, Pod Security Standards, automated TLS, SSO across all tools.

All Infrastructure as Code

Pure Crossplane — no Terraform, no ClickOps. Every resource is a Kubernetes manifest.

Built Entirely with AI

Every line of code, every kubectl command, every debugging session — done through Claude Code across 100+ paired sessions. Zero lines written manually.

3-Tier Bootstrap Flow

The platform bootstraps itself through three tiers, progressively upgrading from static credentials to fully keyless authentication.

1

Bootstrap Cluster (Local)

Kind cluster on a laptop. Crossplane + AWS provider with temporary secret-based credentials. Creates the management cluster, then is deleted.

↓
2

Management Cluster (EKS)

Self-managing EKS cluster. Crossplane with IRSA (AWS) and Workload Identity Federation (GCP) — zero static credentials. Runs ArgoCD, Backstage, Kargo, and the full observability stack.

↓
3

Workload Clusters (EKS + GKE)

Ephemeral clusters provisioned on-demand via Backstage templates. Platform services (Istio, cert-manager, ExternalDNS) auto-deployed by ArgoCD. Kargo promotes applications through staging to production.

Implementation Roadmap

Built iteratively across 10 phases, each expanding the platform's capabilities.

  • Phase 0: Foundation Complete
    • AWS account setup with IAM Identity Center
    • Crossplane bootstrap on Kind cluster
    • CloudTrail audit logging
  • Phase 1: Management Cluster Complete
    • EKS management cluster via Crossplane
    • IRSA for keyless AWS authentication
    • Crossplane pivot from Kind to EKS
  • Phase 2: Multi-Cloud Complete
    • GCP Workload Identity Federation (keyless)
    • Crossplane GCP providers
    • GKE cluster provisioning
  • Phase 3: GitOps Complete
    • ArgoCD app-of-apps pattern
    • Crossplane compositions for EKS + GKE
    • Cluster claim API for self-service provisioning
  • Phase 4: Observability Complete
    • Prometheus + Grafana for metrics
    • Loki for log aggregation
    • Tempo for distributed tracing
  • Phase 5: Security & Policy Complete
    • Kyverno baseline policies (enforce + audit)
    • Pod Security Standards
    • External Secrets Operator + AWS Secrets Manager
  • Phase 6: Ingress & TLS Complete
    • Istio service mesh + Gateway API
    • cert-manager with Let's Encrypt wildcard certificates
    • ExternalDNS for automated DNS management
  • Phase 7: HTTPS & Identity Complete
    • Dex OIDC provider with GitHub identity (enterprise: swap for any IdP)
    • SSO across ArgoCD, Backstage, Grafana, Kargo
    • *.plateng.dev with automated TLS
  • Phase 8: Developer Portal Complete
    • Backstage with software catalog and templates
    • Self-service cluster creation (EKS + GKE)
    • Ephemeral clusters with TTL-based cleanup
  • Phase 9: Progressive Delivery Complete
    • Kargo promotion pipelines (staging → production)
    • Layer separation with ClusterGateway pattern (ADR-0015)
    • E2E validated on both EKS and GKE

Architecture Decisions

Key decisions documented as Architecture Decision Records.

  • ADR-0009 Crossplane IRSA authentication for AWS providers (no static credentials)
  • ADR-0010 ArgoCD annotation+label tracking for safe resource pruning
  • ADR-0011 GCP Workload Identity Federation for keyless cross-cloud access
  • ADR-0012 Istio Gateway API for unified ingress across management and workload clusters
  • ADR-0013 Dex as unified OIDC provider for SSO across all platform tools
  • ADR-0014 Kargo for GitOps-native progressive delivery over Argo Rollouts
  • ADR-0015 ClusterGateway layer separation for independent cluster and ingress lifecycle

Cost-Conscious Decisions

  • Ephemeral workload clusters with TTL-based cleanup (not always-on)
  • Management cluster lifecycle managed for cost (up only when needed)
  • Single management cluster serving both clouds
  • GitHub Pages for status page hosting (free)
  • GitHub Actions for status checks and E2E tests (free for public repos)
